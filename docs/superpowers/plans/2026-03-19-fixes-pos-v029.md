## Task 1 — server.js: excluir CARREGADO e LIBERADO P/ CT-e da programação diária

**Spec:** Issue 2 — `docs/superpowers/specs/2026-03-19-fixes-pos-v029.md`

**Files:**
- Modify: `server.js` (linha 1745-1746)

**Problema:** `gerarProgramacaoDiaria` inclui veículos com `LIBERADO P/ CT-e` e `CARREGADO` na contagem. Esses veículos com `data_prevista` antiga viram reprogramados indevidamente.

- [ ] **Step 1: Localizar a query**

Abrir `server.js` e ir até a função `gerarProgramacaoDiaria` (linha ~1738). O WHERE atual é:
```sql
WHERE (status_recife IS NULL OR status_recife NOT IN ('FINALIZADO', 'Despachado', 'Em Trânsito', 'Entregue'))
  AND (status_moreno IS NULL OR status_moreno NOT IN ('FINALIZADO', 'Despachado', 'Em Trânsito', 'Entregue'))
```

- [ ] **Step 2: Atualizar o WHERE**

Substituir pelas linhas:
```sql
WHERE (status_recife IS NULL OR status_recife NOT IN ('FINALIZADO', 'Despachado', 'Em Trânsito', 'Entregue', 'LIBERADO P/ CT-e', 'CARREGADO'))
  AND (status_moreno IS NULL OR status_moreno NOT IN ('FINALIZADO', 'Despachado', 'Em Trânsito', 'Entregue', 'LIBERADO P/ CT-e', 'CARREGADO'))
```

Nota: a lógica AND mantém o comportamento de exibir veículos com apenas uma unidade em status terminal (ex: `status_recife = 'LIBERADO P/ CT-e'` mas `status_moreno = 'AGUARDANDO'` ainda aparece na programação porque só Recife está no terminal).

## Task 7 — PainelCte.js: filtrar cards CT-e pela unidade que aceitou

**Spec:** Issue 5 — `docs/superpowers/specs/2026-03-19-fixes-pos-v029.md`

**Files:**
- Modify: `src/components/PainelCte.js`

**Problema:** Quando um operador aceita um CT-e pelo sininho, o card aparece nas duas unidades (Recife e Moreno). O campo `unidade_emissao` é gravado no momento do aceite com `user.cidade`, mas o painel não filtra por ele.

**Como funciona:**
- `PainelCte` recebe `ctesRecife` e `ctesMoreno` já pré-filtrados por `origem` em `App.js`
- Mas veículos com múltiplas operações (uma em Recife, outra em Moreno) têm `origem` que pode ser ambíguo
- `unidade_emissao` é o campo correto — é `user.cidade` da pessoa que aceitou o CT-e
- O objeto CTE retornado pelo GET `/ctes-ativos` inclui `unidade_emissao` via `...dados` (spread de `dados_json`)
- `useAuthStore` está disponível — usar `useAuthStore.getState().user?.cidade` para obter a unidade do usuário logado

- [ ] **Step 1: Adicionar import de `useAuthStore`**

No topo de `PainelCte.js`, após as imports existentes:
```js
import useAuthStore from '../store/useAuthStore';
```

(Verificar o caminho correto do store — pode ser `../stores/useAuthStore` ou similar. Conferir como outros componentes importam o store antes de adicionar.)

- [ ] **Step 2: Obter a unidade do usuário logado**

Dentro do componente (logo após as declarações de `isRecife`, `listaCtes`, etc.), adicionar:
```js
const userUnidade = useAuthStore(s => s.user?.cidade);
```

- [ ] **Step 3: Adicionar filtro por `unidade_emissao` no `ctesFiltrados`**

O filtro atual (linhas 31-44) filtra apenas por data. Adicionar a condição de unidade ao final:
```js
const ctesFiltrados = listaCtes.filter(cte => {
    const dataEmissao = cte.data_entrada_cte;
    if (!dataEmissao) return true;

    let dataComparacao;
    if (dataEmissao.includes('/')) {
        const partes = dataEmissao.split('/');
        dataComparacao = `${partes[2]}-${partes[1].padStart(2, '0')}-${partes[0].padStart(2, '0')}`;
    } else {
        dataComparacao = dataEmissao;
    }

    if (!(dataComparacao >= filtroDataInicioCte && dataComparacao <= filtroDataFimCte)) return false;

    // Filtrar por unidade que aceitou o CT-e (retrocompatível: mostrar se campo ausente)
    if (cte.unidade_emissao && userUnidade) {
        return cte.unidade_emissao === userUnidade;
    }

    return true;
});
```
