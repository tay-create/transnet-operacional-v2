# Fixes Pós v0.2.9 — Design Spec

**Data:** 2026-03-19
**Status:** Aprovado

---

## Contexto

Após o release v0.2.9 (Programação Diária v2 + Finalizar Operação), seis issues foram identificados em produção:

1. CT-e Liberado não contabilizado corretamente no dashboard operacional
2. Programação diária inclui CT-e Liberados como reprogramados (errado)
3. Exibição da programação ainda mostra split Recife/Moreno (simplificar)
4. Hover das docas exibe motoristas com status CARREGADO (deve ocultar)
5. Card CT-e aparece nas duas unidades após aceite pelo sininho (deve aparecer só na unidade que aceitou)
6. Números acima das barras no dashboard são pequenos demais (ilegíveis)

---

## Issue 1 — Dashboard: CT-e Liberado não contabilizado

### Causa raiz

O commit v0.2.9 (`fix: botão CT-e não altera status do card, usa flag cte_antecipado`) mudou o critério de contagem do status `LIBERADO P/ CT-E` no dashboard para usar a flag `cte_antecipado_recife`/`cte_antecipado_moreno`. Veículos liberados pelo fluxo padrão (sem `cte_antecipado`) têm o status correto mas a flag `false`, logo não aparecem na barra de status.

### Fix

Em `DashboardTV.js`, no bloco que monta os dados por status para o gráfico de barras, substituir o critério `cte_antecipado_recife/moreno` pelo status real:

```js
// ANTES:
v.cte_antecipado_recife

// DEPOIS:
v.status_recife === 'LIBERADO P/ CT-E'
```

Aplicar o mesmo para Moreno (`status_moreno === 'LIBERADO P/ CT-E'`).

## Issue 2 — Programação: CT-e Liberado vira reprogramado

### Causa raiz

`gerarProgramacaoDiaria` em `server.js` exclui apenas veículos em `FINALIZADO, Despachado, Em Trânsito, Entregue`. Veículos com `LIBERADO P/ CT-E` e `data_prevista` anterior a hoje são incluídos e contados como reprogramados.

### Fix

Adicionar `LIBERADO P/ CT-E` e `CARREGADO` à lista de status terminais/excluídos na query, por unidade:

```sql
WHERE (
  status_recife IS NULL
  OR status_recife NOT IN ('FINALIZADO','Despachado','Em Trânsito','Entregue','LIBERADO P/ CT-E','CARREGADO')
)
AND (
  status_moreno IS NULL
  OR status_moreno NOT IN ('FINALIZADO','Despachado','Em Trânsito','Entregue','LIBERADO P/ CT-E','CARREGADO')
)
```

> **Lógica:** Um veículo com CT-e liberado em uma das unidades mas ainda em aberto na outra continua elegível para aparecer na programação (pois o AND garante que só é excluído quando AMBAS as unidades estão em status terminal).

## Issue 3 — Programação: simplificar exibição (sem split de unidade)

### Mudanças no Frontend (`PainelProgramacao.js`)

#### Cabeçalho da tabela
| OPERAÇÃO | PROGRAMADOS | REPROGRAMADOS |

(Renomear "QUANTIDADE" → "PROGRAMADOS".)

#### Células — novo formato e legado
- **PROGRAMADOS:** `recife + moreno` (soma total, sem label de unidade)
- **REPROGRAMADOS:** `reprogramado_recife + reprogramado_moreno` (soma total); cor vermelha quando > 0

#### Rodapé (TOTAL GERAL)
```
PROGRAMADOS    → total geral (Σ de todos recife + moreno)
REPROGRAMADOS  → total geral reprogramados
```

#### Gráfico de rosca — um único gráfico
Substituir os dois gráficos atuais por um único gráfico de rosca com duas fatias:
- **Programados** (azul `#3b82f6`) = soma total de todos recife + moreno
- **Reprogramados** (vermelho `#dc2626`) = soma total de todos reprogramados

Tooltip do gráfico exibe os valores absolutos de cada fatia.

#### Compatibilidade legado
Snapshots com apenas campo `reprogramado` (sem split): `reprogramado` é usado diretamente como total de reprogramados; `recife + moreno` como total de programados (igual ao comportamento atual).

### Mudanças no Backend (`server.js` — `gerarProgramacaoDiaria`)

Nenhuma mudança necessária na estrutura do JSON gerado (`reprogramado_recife`, `reprogramado_moreno`, `recife`, `moreno`). O frontend simplesmente soma os campos em vez de exibi-los separados. Novos snapshots continuam com o formato v0.2.9.

### Mudanças no `RelatorioImpressao.js`

Mesmo padrão: colunas mostram total (sem split), rodapé mostra total geral. Resumo executivo: `Programados: X | Reprogramados: Y`.

## Issue 4 — Hover das docas: ocultar motoristas CARREGADO

### Causa raiz

O builder do mapa de hover (`docaVeiculoMapR`/`docaVeiculoMapM`) não filtra por status — inclui todos os veículos na doca independente do status.

### Fix

Em `DashboardTV.js`, no forEach que constrói `docaVeiculoMapR` e `docaVeiculoMapM`, adicionar filtro de status antes de mapear:

```js
// Para Recife:
veiculosRecife.forEach(v => {
  if (v.status_recife === 'CARREGADO' || v.status_recife === 'LIBERADO P/ CT-E') return;
  const doca = v.doca_recife;
  if (!doca || doca === 'SELECIONE') return;
  docaVeiculoMapR[doca] = { motorista: v.motorista, coleta: v.coletaRecife || v.coleta || '' };
});

// Para Moreno: idem com status_moreno e doca_moreno
```

## Issue 5 — CT-e em duas unidades após aceite pelo sininho

### Causa raiz

A notificação de CT-e pendente é recebida na fila de ambas as unidades. Quando um operador aceita via sininho, o card CT-e é criado com `unidade_emissao` preenchido, mas o painel CT-e não filtra por unidade ao renderizar — exibe todos os cards com status CT-e aberto.

### Fix

No painel CT-e (componente que lista cards CT-e), filtrar os veículos exibidos por `unidade_emissao === userUnidade` (onde `userUnidade` é a cidade/unidade do usuário logado).

Se `unidade_emissao` for nulo (registros anteriores ao fix), exibir normalmente (compatibilidade).

---

## Issue 6 — Números acima das barras: aumentar tamanho

### Fix

Em `DashboardTV.js`, nos `<text>` customizados que renderizam os labels acima de cada barra (Recife e Moreno), aumentar `fontSize` de `10` para `13`:

```jsx
// ANTES:
fontSize={10}

// DEPOIS:
fontSize={13}
```

Aplicar em ambas as barras (Recife e Moreno).
